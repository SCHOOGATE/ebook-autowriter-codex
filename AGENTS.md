# eBook AutoWriter for Codex v4.0

電子書籍（25,000字・5章構成）を対話型でリサーチ → 執筆 → メタデータ → 表紙 → A+画像 → 申請データまで一括生成。
**画像はすべてChatGPT Images 2.0をChrome DevTools経由で生成する（APIクレジット不要）。**
**要所でユーザー確認を挟み、承認後に自動進行する。**

## 言語

- 日本語で応答する
- プロンプト・技術用語は英語OK

## 起動方法

ユーザーが「**{テーマ}の電子書籍を制作**」と入力して起動する。
（例: 「AI副業の電子書籍を制作」「猫のアロマケアの電子書籍を制作」）

### Chrome DevTools 自動前処理（ユーザー操作名に依存しない）

電子書籍制作・制作再開・表紙生成・A+生成・画像生成テスト・DevTools確認に関係する依頼を受けたら、ユーザーが「テスト開始」「接続確認」等の固定文言を入力していなくても、作業開始前に以下を自動実行する。

```
1. ~/.codex/config.toml の [mcp_servers.chrome-devtools] を確認する
2. args が ["-y", "chrome-devtools-mcp@latest", "--browserUrl", "http://127.0.0.1:9224"] でなければ修正する
3. http://127.0.0.1:9224/json/version を確認する
4. 9224 が未起動なら Chrome book creation 相当のコマンドでChromeを起動する:
   "C:\Program Files\Google\Chrome\Application\chrome.exe" --remote-debugging-port=9224 --remote-allow-origins=* --user-data-dir="%USERPROFILE%\chrome-profiles\profile-9224" https://chatgpt.com
5. http://127.0.0.1:9224/json/list に chatgpt.com タブがあるか確認する
6. chatgpt.com タブがなければ 9224 のChromeで https://chatgpt.com を開く
7. 新しいCodex実行が必要な場合は codex exec --dangerously-bypass-approvals-and-sandbox で mcp__chrome-devtools__list_pages を実行し、MCPが9224側を見ていることを確認する
```

この前処理は、ユーザーに「Chromeを起動してください」「Codexを再起動してください」と依頼して停止する前に必ず実行する。
ただし、ChatGPTログイン画面が出た場合だけは、認証情報を扱わずユーザーに通常Chrome上でログインしてもらう。

テーマは起動プロンプトから自動抽出し、残りの情報を質問する:

```
テーマ「{抽出したテーマ}」で電子書籍の制作を開始します。

以下を教えてください。
1. 著者名（必須）: Kindle出版時の著者名
2. 参考資料（任意）: URL、ファイル、テキストなど
```

回答を受け取ったら、Phase 1 から自動開始する。

※ 起動プロンプトにテーマを含めると、後から何の書籍か判別しやすくなる。

## 最重要ルール

```
1. 「ファイルが存在する」≠「完了」。内容の品質チェックが必須
2. 各Phaseの末尾で python scripts/validate_*.py を必ず実行する
3. 検証NGなら該当Phaseを再実行する（最大2回リトライ）
4. DOCX変換はpython-docxで行う（setup.shで事前インストール済み。pandocは不可）
5. 画像はファイルサイズ50KB以上を必須とする（ダミー禁止）
6. 原稿の同一文反復は禁止（n-gram重複率15%未満を厳守）
7. 原稿はHTML形式で出力する（Markdown原稿とは別にHTMLも生成）
8. 不可視文字を絶対に混入させない（ZWNJ U+200C、Tags U+E0000〜U+E01FF 等）
9. Markdownでは1文（句点「。」まで）= 1段落とする（Kindle読みやすさのため）
10. 画像はPNG/JPGのまま output/{slug}/images/ に直接保存する（変換不要）
11. 表紙画像はJPG形式で出力する（KDP申請可能＋ファイルサイズ削減）
12. 画像生成後はPillowで正しいサイズにリサイズする（表紙1600x2560 / A+970x600）
13. ファイル内容の確認にPowerShellの Get-Content や type コマンドを絶対に使わない（Windows環境ではcp932で読むため日本語が文字化けし、正常なファイルを壊す原因になる）。ファイル確認は必ず python scripts/validate_*.py または python -c "print(open(path, encoding='utf-8').read())" で行う
14. Get-Content の文字化け表示を見て「ファイルが壊れている」「互換ラベルが必要」と判断してファイルを修正する行為は絶対禁止。validateスクリプトがPASSしていればファイルは正常
```

## リトライ時の絶対ルール

```
■ 原稿（manuscript.md / manuscript.html）のリトライ:
  - 不合格の章を特定し、その章を「全文削除してから新しく書き直す」
  - 既存テキストの末尾に追記（append）は絶対禁止
  - 書き直す際は、前の版と異なる具体例・表現・構成を使う
  - リトライ後も n-gram重複率が15%以上なら、原稿全体を1から再生成する

■ メタデータ（listing.txt / kindle_application.txt）のリトライ:
  - listing.txt の紹介文は最低3,000字（不足時は内容を深掘りして加筆）
  - kindle_application.txt の書籍説明文は最低2,500字（PASONA各節を充実させる）
  - 文字数不足で再生成する場合も、同じ文の繰り返しで水増ししない

■ コミットルール（バイナリは絶対にコミットしない）:
  - Phase 3〜5 完了時: テキストファイルのみコミット（manuscript/listing/kindle_app等）
  - Phase 6 完了時: 表紙画像をリサイズして images/ に保存
  - Phase 7 完了時: A+画像4枚をリサイズして images/ に保存
```

## チェックポイントのルール

- ★マークのタイミングでは**必ずユーザーに確認を求めて停止する**
- ユーザーが「OK」「進めて」等で承認したら次へ進む
- ユーザーが修正指示を出したら修正してから再確認
- ★マーク以外のPhaseは確認なしで自動進行する

## Phase 1: 入力受付

起動プロンプト「{テーマ}の電子書籍を制作」からテーマを自動抽出済み。
ユーザーから著者名・参考資料を受け取る。

受け取れる参考資料の形式:
- ファイル（PDF、テキスト、Markdown、DOCX）
- URL（Web検索で内容取得）
- テキスト（直接貼り付け）
- 複数資料の組み合わせ
- なし（テーマのみでもOK）

受け取ったら即座にPhase 2へ進む。

## Phase 2: 5層ディープリサーチ

参考資料からテーマ・キーワードを抽出し、以下の5層リサーチを実行する。

### Layer 1: YouTube専門家の知見
- 「{テーマ} やり方 解説」「{テーマ} プロ 実践」「{テーマ} 2026 最新」等で検索
- 再生回数が多い動画・専門チャンネルを**5〜10本**特定
- 各動画について: タイトル、チャンネル名、要点、独自ノウハウ、具体的手法を記載
- **動画ページのURLを必ず記録する**

### Layer 2: note専門家の記事
- 「{テーマ} site:note.com」で検索
- 上位**5〜10記事**の内容を取得
- 著者の専門性、独自フレームワーク、具体的数値、読者反応を記載
- **記事URLを必ず記録する**

### Layer 3: SNS/ショート動画トレンド
- Instagram、TikTokのバズキーワード・切り口を調査
- インフルエンサーの推しポイント、Z世代に響く表現を抽出
- **バズワード3つ以上**を明記

### Layer 4: 市場・競合・書籍分析
- Amazon上位書籍**3冊以上**の目次構成・レビュー分析
- 星1-2レビューから読者の不満・期待を抽出
- 競合にない切り口・空白地帯を特定
- **書籍名とURLを必ず記録する**

### Layer 5: 読者の悩み・ニーズ
- Yahoo知恵袋、Q&Aサイトで**10件以上**の悩みを収集
- 初心者がぶつかる壁、「こういう本があれば」という要望を抽出
- **各悩みの出典URLを記録する**

### research.md の必須構造

`templates/research_template.md` を参照し、以下の構造で出力すること:

```markdown
# リサーチ結果: {テーマ}

## 参考資料の要約
{受け取った資料のポイント整理}

{整理}

## 品質基準チェック

- note: {N}記事 ✅/❌
- SNSバズワード: {N}語 ✅/❌
```

### リサーチURL品質ルール
- Amazon書籍は個別商品ページURL必須（`/dp/` or `/gp/` を含むURL）。検索URL（`/s?k=...`）は不可
- Layer 1のYouTube動画はyoutube.com/watch?v= のURL必須。note記事URLをYouTube動画として記載しない
- 各LayerのURLはそのLayerの種別に合致すること（YouTube→動画URL、note→記事URL、Amazon→商品ページURL）

### Phase 2 完了時の検証（必須）

```bash
python scripts/validate_research.py output/{slug}
```

**検証基準:**
- research.md が3,000字以上
- 5層すべての見出しが存在
- 参考URL 15件以上
- 品質基準チェックセクションが存在

**NGの場合:** 不足しているLayerのリサーチを追加実行し、research.mdを更新してから再検証。

リサーチ完了後、**ユーザーに確認を求めて停止する。**

### 提示する内容

1. リサーチ結果のサマリー（各Layer の要点を3行程度ずつ）
2. 売れるタイトル/サブタイトル案を**10個**生成（以下の型を必ず混ぜる）

| # | 型 | 例 |
|---|---|---|
| 1 | ベネフィット直球型 | 「3日で身につく〇〇」 |
| 2 | 数字提示型 | 「7つの〇〇ルール」 |
| 3 | 問いかけ型 | 「なぜ〇〇は××なのか？」 |
| 4 | 否定・常識破り型 | 「〇〇をやめなさい」 |
| 5 | ターゲット限定型 | 「初心者のための〇〇入門」 |
| 6 | ストーリー型 | 「ゼロから〇〇になった話」 |
| 7 | 比較・選択型 | 「AとBどっちがいい？」 |
| 8 | 完全ガイド型 | 「〇〇完全攻略ガイド」 |

| 10 | 権威・最新型 | 「2026年最新版 〇〇のすべて」 |

3. ユーザーに確定してもらう:
   - タイトル + サブタイトル（番号選択 or 改変 or 新規）

## ★確認2: 構成（目次）確認

タイトル確定後、目次を設計してユーザーに提示する。

### 提示する目次

**書籍全体の感情設計（大アーク）を意識した構成にする。** 各章にはPASCAL的な感情的役割を割り当て、読者の感情が「共感→危機感→希望→自信→自立」と段階的に変化する設計にする。

```
書籍タイトル: {確定タイトル}
想定読者: {テーマから推定}
読者のゴール: {この本を読んで何ができるようになるか}
大アーク: {共感→危機感→希望→自信→自立}（書籍全体の感情遷移）

はじめに（1,500〜2,000字）【8%】感情: 共感＋約束
  - フック: 読者の「あるある」な日常シーンで始める
  - 読者への約束: 「この本を読むとあなたは○○になれます」
  - 本書の使い方（どの章から読んでもOKの案内）

第1章: {章タイトル}（3,500〜4,500字）【16%】感情役割: Problem（問題の明確化）
  → 読者が「やっぱりそうだったんだ」と感じる章
  小見出しの数・構成はテーマに合わせて自由に設計（2〜5節）

第2章: {章タイトル}（4,000〜5,000字）【18%】感情役割: Agitation（深掘り・原因解明）
  → 読者が「このままだとまずい」→「変わりたい」と感じる章
  小見出しの数・構成はテーマに合わせて自由に設計（2〜5節）

第3章: {章タイトル}（5,500〜7,000字）【26%】感情役割: Solution（解決策の核心）★最大分量
  → 読者が「なるほど、こうすればいいのか」と感じる章
  → 本書の最も価値あるコンテンツをここに集中させる
  3.1〜3.5（同形式。他章より各節が長い）

第4章: {章タイトル}（4,500〜5,500字）【20%】感情役割: Credibility + Action（実践・信頼構築）
  → 読者が「自分にもできそうだ」と感じる章
  → ケーススタディ・具体的手順・裏付けデータを厚く
  4.1〜4.5（同形式）

第5章: {章タイトル}（3,000〜4,000字）【12%】感情役割: Action発展（応用・自走）
  → 読者が「もっとやりたい」と感じる章
  → 応用編・発展的な内容。軽く終わらせて余韻を残す
  5.1〜5.5（同形式）

おわりに（1,500〜2,000字）【8%】感情役割: Lead（読者の未来を描く）
  - 全章の学びを統合する1文
  - 読者が「読んでよかった」と感じる締めくくり
  - 次のアクション（別の本、コミュニティ、実践の継続等）
```

### 各章内の感情設計（節の数はテーマに応じて2〜5節）

**各章の中でも読者の感情を自然に動かす。** 節の数は固定しない。テーマに必要な分だけ設ける。

各節には以下の役割から適切なものを割り当てる（全部使う必要はない）:

```
・共感フック — 読者の日常シーンで問題を描写
・原因解明 — なぜそうなるのか、因果を説明
・解決策 — 具体的な方法論
・実践例 — 読者像別のケーススタディ
・即実践 — 今日からできるアクション + 読者ワーク
```

**必須なのは「章の冒頭で共感を作り、章の末尾で次の行動が見える」こと。** 中間の節数と役割は自由。

**禁止:** 全章が同じ節数・同じ役割構成になること。第1章が3節なら第3章は4節、のように章ごとに構成を変える。

### 構成ルール
- **章の順序: Problem → Agitation → Solution → Action → 応用**（感情設計に従う）
- 各章は独立して読んでも価値がある
- 章をまたいで内容が行き来しない
- 読者が「次に何をすればいいか」がわかる
- 各章の小見出し（###）は**2〜5節**。テーマに応じて適切な数にする（5節強制は禁止。3節で十分なら3節にする）
- **第3章（Solution）が最大分量**。ここに本書の核心を集中させる
- 小見出し番号は必ず {章番号}.1 から始まる連番にする（例: 第4章なら 4.1→4.2→4.3）。最初の節は必ず .1 で始めること。2.2から始まる（2.1がない）、4.2から始まる（4.1がない）のような先頭欠落も禁止。4.1→4.4→4.5のような飛び番も禁止。節を増減した場合は必ず .1 から振り直す
- 小見出し（###）の直後は必ずそのテーマに即した具体的な本文で始めること。フィラー・前節の続き・別テーマの内容を置くことは禁止
- 章末の「次章への引き」（フック）の後に、補足テキスト・フィラー・追加段落を置かない。フックで章を終了すること

**ユーザーが「OK」と承認したら、Phase 2.5〜5 を一気に自動実行する。**

## Phase 2.5: 章別設計書（★元SKILLs超え品質の核心）

**構成が承認されたら、執筆前に必ず章別設計書を生成する。確認なしで自動進行。**

リサーチ結果（research.md）を「どの章の何節で使うか」具体的に配分し、章間の重複を防止する。
この設計書がないまま執筆すると、抽象的な文章の反復になる。**設計書なしの執筆開始は禁止。**

### 生成する内容（output/{slug}/chapter_blueprint.md）

各章について以下を記述する:


### 設計書の品質基準

- research.md の5層すべてのLayerから最低1箇所ずつ引用先を指定していること
- Layer 5（読者の悩み）10件以上のうち、8件以上がいずれかの章・節に対応していること
- 「この章では扱わない内容」が全章で矛盾なく整合していること（Aで回すと書いた内容がBで実際に扱われている）
- 同じ具体例・データが複数の章に重複配分されていないこと
- 各章のフックが次章の導入と論理的につながっていること

## Phase 3: 原稿執筆（25,000字・HTML + Markdown）

**構成が承認されたら、Phase 3〜5 は確認なしで一気に自動実行する。**

### 生成フロー（モデル別）

**GPT-5.5使用時: 一括生成（推奨）**

GPT-5.5は128Kトークン出力に対応しており、25,000字の原稿を1回で生成できる。
chapter_blueprint.md の全章設計を参照し、はじめに〜おわりに を通しで執筆する。

Step 1: 原稿全体を一括生成（manuscript.md + manuscript.html）
Step 2: validate_manuscript.py で全体検証
Step 3: FAILの場合 → エラー箇所を修正して再検証（全体再生成ではなく部分修正）

一括生成の利点:
- 章間の整合性がモデル側で保証される
- フィラーで文字数を稼ぐ必要がない（全体を見て配分できる）
- 章間の具体例・データの重複を自動回避できる

**GPT-4o以前のモデル使用時: 分割生成（フォールバック）**

出力トークン制限（16K）のため、章ごとに分割生成する。

Step 1: はじめに → validate → PASS後に次へ
Step 2: 第1章 → validate → PASS後に次へ
Step 3: 第2章 → validate → PASS後に次へ
Step 4: 第3章 → validate → PASS後に次へ
Step 5: 第4章 → validate → PASS後に次へ
Step 6: 第5章 → validate → PASS後に次へ
Step 7: おわりに → validate → PASS後に次へ
Step 8: 全文結合（manuscript.md + manuscript.html）→ validate_manuscript.py で全体検証

**共通: 各章での検証項目**
- 文字数が章の最低基準を満たしているか
- フィラー行が0であるか（テンプレフィラー自動検出 v4.5）
- 小見出しの直後に具体的な本文があるか
- chapter_blueprint.md で指定した具体例・データが使われているか
- research.md のURLまたはドメインが原稿中に引用されているか

文字数が足りない場合の対処:
- フィラーや水増しテキストで埋めることは絶対禁止
- chapter_blueprint.md に戻り、以下を追加して再執筆する:
  1. research.md から未使用の専門家データ・具体例を追加配分
  2. 読者像に合わせた「あなた」語りのシナリオを詳細に展開（600字→900字に拡大）
  3. 読者ワークの設問を増やす
  4. リサーチ引用ストーリー化を追加

### 出力形式

原稿は**2つの形式**で出力する:

1. `manuscript.md` — Markdown形式（検証・変換用）
2. `manuscript.html` — HTML形式（最終成果物）

### HTML装飾ルール（Kindle申請品質）

1. **改ページ**: 章（H2）と節（H3）に `page-break-before: always` を適用済み
2. **太字**: 重要な用語・キーワードは `<strong>` で囲む（各段落に1〜2箇所）
3. **強調**: 補足的な強調は `<em>` を使用
4. **ポイントボックス**: 各章に1〜2箇所、`<div class="point">` で重要ポイントを囲む
5. **補足ボックス**: 注意事項は `<div class="note">` で囲む
6. **目次リンク**: 各章のIDに `<a href="#ch1">` でジャンプ可能にする
7. **タイトルページ**: 書籍タイトル・サブタイトル・著者名を中央配置
8. **箇条書き**: `<ul>/<ol>` で構造化
9. **段落インデント**: `text-indent: 1em` で日本語書籍の体裁
10. **句点で改行しない**: HTML版では句点改行は行わない（DOCX版のみの処理）

### 執筆ルール（厳守）

- **総文字数: 25,000字以上**（25,000字未満は不合格）
- **文字数傾斜配分**（全章均等は禁止。第3章に最大リソースを集中）:
  - はじめに: 1,500〜2,000字【8%】（1,200字未満は不合格）
  - 第1章（Problem）: 3,500〜4,500字【16%】（3,000字未満は不合格）
  - 第2章（Agitation）: 4,000〜5,000字【18%】（3,500字未満は不合格）
  - 第3章（Solution）: 5,500〜7,000字【26%】★最大（5,000字未満は不合格）
  - 第4章（実践）: 4,500〜5,500字【20%】（4,000字未満は不合格）
  - 第5章（応用）: 3,000〜4,000字【12%】（2,500字未満は不合格）
  - おわりに: 1,500〜2,000字【8%】（1,200字未満は不合格）
- 文体: **です・ます調で統一**（80%以上）
  - NG: 「〜だ」「〜である」「〜しよう」「〜だろう」
  - OK: 「〜です」「〜になります」「〜してみましょう」「〜してください」
- 段落: 3〜4文ごとに改行
- 具体例: 各章に最低2つの具体例・事例
- **表（テーブル）禁止**: 比較情報は箇条書きで表現
- **コードブロック禁止**: コマンドは通常テキストとして記述
- **ASCII図禁止**: 罫線文字は使わない
- **ユーザー名禁止**: 「読者の皆さん」等の一般表現を使用
- **著者情報禁止**: 原稿本文に著者名・プロフィールを入れない

### 一次情報・ストーリーの必須要件（各章25%以上）

電子書籍は匿名著者（コミュニティメンバーが自分名義で出版）が多いため、「私の体験」の代わりに**3種類の一次情報パターン**を使い分ける。

**パターンA: 「あなた」＋属性描写シナリオ（メイン・各章に必須）**
```
「もしあなたが30代の事務職で、毎朝7時に出社し、夕方には疲労で集中力が切れているなら…
通勤電車の中で○○を試してみてください…」

「週末だけランニングをしている方なら、月曜の外回りで脚が重くなった経験があるかもしれません…」
```
- 架空の人名は使わない。「あなた」「〜の方なら」「〜の場合は」で語りかける
- chapter_blueprint.md で設計した3〜5パターンの読者像を使う
- 各章に1つ、異なる読者像に向けたシナリオを**600字以上**で展開
- 第1章で提示した読者像Aの問題が、第4章で解決される**物語の回収**を入れる
- 読者像はLayer5（読者の悩み）から具体的に設計する

**パターンB: リサーチ引用ストーリー化（各章1〜2箇所）**
```
「YouTube専門家の○○氏は、10年間で3,000人のクライアントを見てきた中で、
最も多い失敗パターンは△△だと語っています。あるクライアントは…」
```
- research.md のLayer1〜4の情報を「誰かの体験談」として再構成する
- 各章に**300字以上**

**リサーチ引用の必須ルール:**
- 各章に最低1件、research.md の専門家名・データ・URLを具体的に引用すること
- 「〜と言われています」「〜とされています」ではなく、「YouTube専門家の○○氏は」「noteの○○記事によると」「ACSMのガイドラインでは」のように出典を明記する
- research.md に60件のURLがあるのに原稿内で引用0件はFAIL基準

**パターンC: 読者ワーク（各章末に1つ）**
```
「ここで、あなた自身の先月の手帳を開いてみてください。
書いてある予定のうち、自分の目標に直結するものは何個ありますか？
3個以下なら、次の節で紹介する方法が特に効果的です。」
```
- 読者自身の体験を振り返らせる問いかけ型
- 各章末に**200〜300字**
- 次の節や次章の内容への橋渡しにもなる

**文字数目安:** パターンA(600字)+B(300字)+C(250字)= 各章1,150字以上 ≒ 各章の25%以上

### 禁止表現リスト（検出時は書き直し必須）

以下の表現はAIが生成しがちな低品質パターン。検出したら具体的な内容に書き換える。

**曖昧な形容詞・副詞（具体性ゼロ）:**
- ❌「〜が大切です」「〜が重要です」「〜がポイントです」
  → ✅ **なぜ大切かの具体理由を書く**（「睡眠が大切です」→「6時間未満の睡眠が3日続くと、判断力が飲酒時と同等まで低下します」）
- ❌「さまざまな」「いろいろな」「多くの」
  → ✅ **具体的な数字・事例を挙げる**（「さまざまな方法」→「代表的な3つの方法」）
- ❌「〜と言われています」「〜とされています」
  → ✅ **誰が言っているか明記する**。research.mdの専門家名・出典を引用する
- ❌「〜かもしれません」「〜の可能性があります」
  → ✅ 断定できる内容は断定する。できないなら条件を明示する（「○○の場合は△△になります」）

**接続詞の連発（論理構造の欠如）:**
- ❌ 同じ接続詞の3連続使用（「また」「そして」「さらに」「加えて」）
  → ✅ 接続詞に頼らず、**文と文の因果関係で自然につなぐ**

**水増し表現（文字数稼ぎ）:**
- ❌ 同じ主張を言い回しを変えて繰り返す（「つまり〜ということです」「言い換えれば〜」が2回以上）
- ❌ 前の段落の要約から始まる段落（「このように〜」で始めて同じ内容を繰り返す）
  → ✅ 各段落は必ず**新しい情報**を1つ以上含む

**AIっぽい文体パターン（禁止）:**
- ❌ 全ての文が「○○ではありません。△△です。」の否定→肯定パターン
  → ✅ 否定→肯定は1章に1〜2回に留める。他は直接的に主張する
- ❌ 全段落の冒頭が「ここで必要なのは」「だから」「つまり」「そのため」
  → ✅ 段落冒頭は具体的な事実・シーン・数字で始める。接続詞で始めない
- ❌ 「〜することが大切です」「〜を意識しましょう」で終わる段落が連続
  → ✅ 読者の行動・判断を具体的に描写して終わる（「来週の月曜に15分だけ試してください」）
- ❌ 「いかがでしたか」「まとめると」「以上が〜です」
  → ✅ 使わない。まとめは読者の次のアクションで締める

**テンプレート型フィラー（水増し）の絶対禁止:**
- ❌「{章名}の補足Nでは、{単語}を見直し…」
- ❌ 同一テンプレートに変数（入浴/睡眠/食事/水分…）を入れ替えて複数行を生成する行為
- ❌ 章末や節末に意味のない定型文を並べて文字数を稼ぐ行為
- → 検出された場合、validate_manuscript.py が自動でFAILとなり、該当章の全文再生成が必要
- → フィラーで25,000字に到達してもPASSしない。実質的な本文のみで文字数を満たすこと

**推奨表現パターン（人間が書く本のように）:**
- 抽象→具体: 「運動は健康に良い」→「朝の15分ウォーキングを始めて2週間、午前中の眠気が消えたという報告は多い」
- 一般論→読者シーン: 「時間管理が必要です」→「月曜の朝、メールを開いた瞬間にその日の予定が崩れた経験はないでしょうか」
- 結論先→理由後: 「結論から言うと、○○です。なぜなら〜」の構文を積極的に使う
- **具体→抽象**: 「田中さんは火曜と木曜に20分走り、日曜に長めの練習を入れた。3ヶ月後にはこの習慣が生活の一部になっていた。時間は作るものではなく、生活の中に見つけるものだ。」 ← 具体的な話から教訓を導く流れが最も自然

### 読者の声ドリブン執筆（各章の必須チェック）

chapter_blueprint.md の「読者の悩み対応」に基づき、**各章の執筆中に以下を必ず確認する**:

1. **悩み直結**: この章は research.md Layer5 のどの悩みに答えているか？（chapter_blueprint.mdの対応番号を参照）
2. **共感シーン**: 読者が「それ、私のことだ」と感じる具体的な日常シーン描写が章の冒頭300字以内にあるか？
3. **因果関係**: 「なぜそうなるのか」の原因→結果の説明があるか？（「○○すると→△△になります。なぜなら〜」）
4. **即実践アクション**: 読者が今日から実践できる具体的なステップ（3ステップ以内）が章内に1つ以上あるか？
5. **裏付け**: Layer1〜4の専門家意見・データ・事例からの裏付けが章内に2つ以上あるか？

### 章間コヒーレンス（1冊の本としての一貫性）

Codex版は逐次執筆のため、この一貫性チェックが元SKILLs（並列執筆）を超える最大の武器。

**各章の執筆完了後、次章の執筆前に必ず確認:**
1. 前章の結論と次章の導入が論理的につながっているか
2. chapter_blueprint.md のフック（前章）が次章の内容と一致しているか
3. 前章で使った具体例・データと同じものを次章で重複使用していないか

**全章執筆完了後の全体検証:**
- 第1章の導入テーマが「おわりに」で回収されているか（円環構造）
- 各章末尾の「次章では〜」が実際の次章内容と一致しているか
- 全5章を通して、読者の知識・スキルが段階的に積み上がる構造になっているか

### 意味的重複の自己チェック（各節の執筆完了時）

n-gram重複率（15%未満）は文字列レベルのチェック。以下は**意味レベル**の重複防止:

**各節（###）の執筆完了時に、以下を自問する:**
1. 「この節の主張を1文で要約すると？」
2. 既に執筆済みの全節の主張と比較する
3. 同じ主張の節があれば、**片方を削除して別の内容に書き換える**

**よくある意味的重複パターン（禁止）:**
- 「第1章で基礎を説明 → 第3章で同じ基礎を別の言葉で再説明」
- 「節1.3で事例A → 節3.2で事例Aの別側面を説明（読者には同じ話に見える）」
- 「各章の冒頭が毎回同じ構造（問いかけ→定義→重要性の説明）で始まる」

### 節末定型ブロックの禁止（★テスト実績: n-gram重複率55.9%の原因）

**各節を同じ「まとめ段落」で締めくくることは絶対禁止。** テスト（marathon-six-months）で以下の定型文が全節末に挿入され、重複率55.9%になった:

```
NG パターン1（全節末に同じまとめ段落）:
「完走を目指す読者にとって、無理をしない判断は消極的な選択ではなく…」
「一週間単位で見れば予定外の日は必ず出ますが…」
「初心者は一度の練習で答えを出そうとしますが…」

NG パターン2（「視点を加えてください」定型フィラー）:
「○○を考える時は、△△を手がかりにして、□□という視点を加えてください。」
→ この文が各節末に10〜20行連続で挿入される。文字数稼ぎのフィラー

NG パターン3（「流れに置くと」テンプレート接続文）:
「この考え方を○○の流れに置くと、△△は単なる知識ではなく…」
→ 節と章テーマを無理に接続するテンプレート文

NG パターン4（「場面で効いてきます」テンプレート末尾）:
「特に「○○」の場面で効いてきます」
→ 全文末に付着するテンプレート残骸
```

**根本原因:** Codexが文字数基準をクリアするために、節の本文が終わった後にフィラーテキストを大量挿入している。**フィラーの挿入は文字数に含めない。フィラーを除去した後の本文が字数基準を満たす必要がある。**

> **モデル推奨:** GPT-5.5を使用すること。GPT-4oは指示遵守率が低く（40%）、フィラー生成の根本原因となる。GPT-5.5は指示遵守率が大幅に向上し、フィラー抑制効果が期待できる。

**正しい節の終わり方:**
- 節1（共感フック）: 読者の共感で終わる。まとめ不要
- 節2（原因解明）: 原因の結論で終わる。まとめ不要
- 節3（解決策）: 解決策の要点で終わる。まとめ不要
- 節4（実践例）: ケーススタディの結果で終わる。まとめ不要
- 節5（即実践）: 読者への問いかけ（パターンC: 読者ワーク）で終わる ← ここだけまとめ的

**各節は、その節固有の結論で自然に終わる。** 「この考え方を○○の流れに置くと」のような接続文で無理に章テーマに戻す必要はない。次の節の見出しが自然なつなぎになる。

**validate_manuscript.py の n-gram重複率チェック（15%未満）で自動検出される。** 15%を超えた場合、まず節末の定型ブロックを探して削除する。

### 章頭プレビューテンプレの禁止（★テスト実績: eigo-kentei フィラー比率51.1%の原因）

**各章の冒頭に、節ごとの「プレビュー」を定型文で並べることは絶対禁止。**

```
NG パターン5（章頭プレビューテンプレ）:
「第1章の実践視点1では、つまずきの原因を読者自身の受験級に置き換えて考えます。」
「第1章の学習場面1では、ただ理解したつもりで終えず、次に解く問題で確認するところまで進めます。」
「第1章の記録方法1では、今日の失敗を責める言葉ではなく、明日の練習名として短く残します。」
「第1章の行動設計1では、十五分で終わる量に絞り、終わったあとに小さな達成感が残るようにします。」
→ このブロックが節の数だけ繰り返される（第3章で5節=20行のフィラー）
```

**章の冒頭は、その章全体の問題提起か具体的エピソードで始めること。**

### 節タイトル差し込みテンプレの禁止（★テスト実績: eigo-kentei 162出現）

**各節の冒頭に、節タイトルの文字列を機械的に差し込んだテンプレートは絶対禁止。**

```
NG パターン6（節タイトル差し込みテンプレ）:
「{節タイトル}では、○○を孤立した作業にせず、合格点へ近づくための材料として扱います。」
「具体的には、{節タイトル}に取り組む前に、今日の到達点を一つだけ決めてから始めます。」
「学習後は、{節タイトル}の結果を感覚で終わらせず、○○します。」
「{節タイトル}の流れを使うと、短い勉強時間でも次の日に続く手がかりが残ります。」
→ 節タイトルを文中に3回以上そのまま引用する構造
```

**各節は、その節固有のエピソード・データ・問いかけで始めること。節タイトルの文字列を本文中にそのまま引用してはならない。**

### 過剰空行の禁止

**連続空行は2行まで。3行以上の連続空行は禁止。** テスト（eigo-kentei）で7連続空行が44箇所検出された。

**validate_manuscript.py v4.5 でパターン5/6/過剰空行を自動検出する。** フィラー除去後の実質文字数が25,000字未満の場合もFAILになる。

### 句点改行ルール + 1文の長さ（Kindle品質）

**1文=1段落（Markdown上で句点「。」ごとに改行＋空行）はそのまま維持する。** Kindle端末での読みやすさに直結するため。

**ただし、1文自体を十分に長くすること。** 短い文が連続するとAIっぽい箇条書きに見える。接続表現を使って1文に2〜3の要素を盛り込み、読み応えのある文にする。

```
NG（1文が短すぎる — 箇条書き的）:
AI副業を学ぶ前に確認してください。

使える時間は週に何時間あるのか。

初期費用はいくらまでなら出せるのか。

OK（1文が長く、接続表現で要素をつなぐ）:
AI副業を学ぶ前に、使える時間は週に何時間あるのか、初期費用はいくらまでなら出せるのかを確認してください。

この2つが見えるだけで最初に取るべき行動が変わるため、ノートに書き出すだけでも価値があります。

朝の30分と夜の30分では集中力が違うので、自分が楽に始められる時間帯を1週間だけ観察してみてください。
```

**1文の長さの目安:**
- 短い文: 30〜50字（テンポを変えたい時、強調したい時に使う。全体の20%以下）
- 標準の文: 60〜100字（メイン。接続表現で2〜3要素をつなぐ）
- 長い文: 100〜150字（複雑な因果関係や具体例を1文にまとめる時）

**接続表現の活用（1文を自然に長くする手段）:**
- 理由: 「〜ため」「〜ので」「〜から」
- 逆接: 「〜が」「〜ものの」「〜にもかかわらず」
- 並列: 「〜だけでなく」「〜に加えて」「〜と同時に」
- 条件: 「〜なら」「〜場合は」「〜であれば」
- 例示: 「たとえば〜のように」「〜といった」

**禁止:** 20字以下の文が3回以上連続すること（箇条書き化のサイン）。

### 不可視文字の混入禁止

**テキスト内にゼロ幅非結合子（U+200C）、Tags（U+E0000〜U+E01FF）、その他の不可視制御文字を絶対に入れない。**
これらが混入するとKindle変換時に「??」として表示される重大な品質問題を引き起こす。

原稿完成後に以下のクリーニングスクリプトを実行すること:

```bash
python scripts/clean_invisible.py output/{slug}
```

### 同一文反復の絶対禁止

**前回の失敗: 見た目は25,000字だが同じ文章の繰り返しだった。**

以下を厳守:
- 同じ文を2回以上使わない
- 同じ段落パターンを繰り返さない
- 各節で異なる具体例・データ・視点を提供する
- validate_manuscript.py の n-gram重複チェック（15%未満）をパスすること

### Markdown版の改ページ・整形ルール

- 各章（`##`）の直前に `\newpage`
- 各節（`###`）の直前に `\newpage`
- はじめに・おわりにの直前にも `\newpage`
- 見出しの前後に空行1行
- 段落間に空行1行

### 見出しレベル（Markdown / HTML共通）

- H1: 書籍タイトル（冒頭1回のみ）
- H2: 章タイトル（はじめに、第1章〜第5章、おわりに）
- H3: 節タイトル（1.1, 1.2, ...）
- H4: 小見出し（必要に応じて）

### 章ごとの文字数確認

各章を書き終えたら、その章の文字数を数え、3,500字未満なら加筆する。

### Phase 3 完了時の検証（必須）

```bash
python scripts/validate_manuscript.py output/{slug}
```

**検証基準（v4.5）:**
- 総字数 >= 25,000
- **フィラー除去後の実質文字数 >= 25,000**（テンプレフィラー自動検出）

- **テンプレフィラー 0行**（章頭プレビュー/節タイトル差し込み/補足N型）

### テンプレート残骸・指示文の混入禁止（★致命的品質バグ）

**過去のテスト（time-management-goal-setting）で発生した実例:**
```
NG（テンプレ残骸が全文末に混入）:
「必要なのは、作業を増やすことではなく、目標との関係が強い行動を先に置くことです、第1章1節の実践5として確認します。」

OK（読者が読む文章のみ）:
「必要なのは、作業を増やすことではなく、目標との関係が強い行動を先に置くことです。」
```

**原稿に以下のパターンが1箇所でもあれば不合格:**
- `第N章N節の実践Nとして確認します`
- `Phase N`, `Step N`, `Layer N` 等の工程名
- `{テーマ}`, `{章タイトル}`, `{slug}` 等のプレースホルダ
- `TODO`, `FIXME`, `PLACEHOLDER`, `INSERT HERE`
- `以下を確認する`, `以下を実行する` 等のAGENTS.md内の指示文
- `YAML`, `validate`, `output/` 等の技術用語
- `サブエージェント`, `エージェント`, `リサーチ結果` 等の制作プロセス用語
- `chapter_blueprint`, `research.md`, `book_meta.md` 等のファイル名
- `Layer 1〜5` 等のリサーチ層の参照

**原稿は「読者が読む本」であり、「AIへの指示書」ではない。**
制作プロセスの痕跡が1文字でも残っていれば、読者にとって意味不明なノイズになる。

**検出した場合の対処:**
1. 混入箇所を含む章を特定する
2. 該当章を全文削除して、読者向けの文章として一から書き直す
3. 書き直し後、再度 validate_manuscript.py を実行する
4. **部分置換（sed的な一括削除）は禁止** — 残骸が混入した章は文章構造自体が破綻しているため、パッチではなく書き直しが必要

**予防策（執筆中に守ること）:**
- AGENTS.md の指示文・テンプレートをそのまま原稿にコピーしない
- 「{変数名}」のような記法は原稿内で使用しない
- 各節を書き終えたら、その節を「読者として音読」し、制作プロセスの痕跡がないか確認する
- 「〜として確認します」「〜を実行します」のような制作者目線の文末は読者目線に変換する

**NGの場合:** 「リトライ時の絶対ルール」に従い、不合格の章を全文削除してから新しく書き直す。既存テキストへのappend禁止。

## Phase 4: 出版メタデータ生成

**確認なしで自動実行（Phase 3 から連続）**

### 手順

1. manuscript.md からテーマ・ターゲット・キーコンセプトを抽出
2. Web検索でキーワードリサーチ
3. メタデータを生成して `listing.txt` に出力

### 生成内容

`templates/listing_template.txt` を参照し、以下を含むこと:

#### タイトル・副タイトル（3案）

```
【提案N：コンセプト】
タイトル: {30文字以内、メインキーワード必須}
副タイトル: {関連キーワード×接続 + ベネフィット文}
```

最重要ルール: メインキーワードは**必ずタイトルに入れる**。

#### 著者名

**book_meta.md で確定した著者名をそのまま使う。** 独自のペンネーム（「○○活用研究室」等）に勝手に変更しない。
listing.txt の著者名 = book_meta.md の著者名 = kindle_application.txt の著者名 が完全一致すること。

#### キーワード（7行 × 各50文字以内）

- タイトルに含まれるKWは除外
- 各行内は半角スペース区切り

#### フリガナ

タイトル・副タイトル・著者名のカタカナ・ローマ字を生成。

#### 紹介文（3,000〜4,000文字厳守 / 4タイプ自動選択）

- emoji禁止、装飾記号禁止、HTMLタグ禁止
- プレーンテキストのみ
- 目次を含める（SEO対策）
- **3,000字未満の場合は必ず加筆して3,000字以上にすること**
- **同一段落・同一文の反復で文字数を稼ぐことは絶対禁止**（3回以上の反復検出で不合格）。文字数が足りない場合は、各タイプの①〜⑥の各パートを深掘りして加筆する
- 原稿のトーンから以下の4タイプを自動選択し、**型に沿って構成する**:

**タイプA: おだやか共感型**（健康/暮らし/自己改善/生活改善/やさしいトーン）
```
① 導入（200字）: 読者の日常の悩みを穏やかに描写する。共感ファースト
② 共感（300字）: 「その気持ち、よくわかります」——読者の感情に寄り添う
③ 本書の約束（400字）: 大きな変革ではなく、小さな変化から始める提案
④ 目次紹介（800字）: 各章の「読者にとっての価値」を記載（目次そのものではなく、各章を読むと何が変わるかを説明）
```

## 制約事項

- 表（テーブル）を原稿内で使わない
- コードブロックを原稿内で使わない
- 著者情報を原稿本文に入れない
- **表紙に著者名を必ず入れる**（book_meta.mdの確定著者名を使用。底部にサブタイトルより小さく配置）
- ASCII罫線図を使わない
- **pip install はエージェントフェーズで実行しない（setup.shで事前インストール済み）**
- **pandoc は使用しない**
- **検証スクリプトはPython標準ライブラリのみで動作する**
- **ファイルのエンコーディングはUTF-8を厳守する**:
  - Pythonでファイルを書く場合: `open(path, 'w', encoding='utf-8')` を必ず指定
  - ファイル冒頭にBOM（`\ufeff`）を入れない
  - **ファイル内容の確認にPowerShellの `Get-Content` や `type` コマンドを使わない**（Windows環境ではcp932で読むため日本語が文字化けする）。代わりにPythonで `open(path, encoding='utf-8').read()` で確認するか、validateスクリプトを使う
  - `Get-Content` の文字化けを見て「ファイルが壊れている」と判断してはならない。validateスクリプトがPASSしていればファイルは正常
- **表紙画像はJPG形式で出力する（PNG生成時はJPGに変換）**
- **バイナリファイル（.png .jpg .jpeg .docx）を直接 git add / git commit してはならない**
- **画像はPNG/JPGのままimages/に保存。Base64変換・encode_binaries・decode_binariesは使わない**
- **画像内の日本語テキストはChatGPT Images 2.0のプロンプトに含めて描画する（Pillow/PILでのテキスト描画は禁止。Codex環境に日本語フォントがない）**
- **全画像（表紙・A+）はHTML/CSS/SVGで描画しない。必ずChatGPT Images 2.0（ブラウザ版 chatgpt.com）でChrome DevTools MCP経由で生成する**
- **DALL-E 3での生成は禁止。生成前後でモデルがGPT-5.5であることを必ず確認する**
- **imagegen skill / gpt-image-2 API での画像生成は絶対に禁止。品質が低いため使用不可。すべての画像はChrome DevTools MCP経由のChatGPT Images 2.0で生成すること**
- **Codexに `image_gen` / `imagegen` / 画像生成スキルが利用可能でも、Phase 6・Phase 7では絶対に使用しない。画像生成スキルの指示より本AGENTS.mdを優先し、必ずChrome DevTools MCPでブラウザ版ChatGPT Images 2.0を操作する**
- **Phase 6・Phase 7の開始時は、最初のツール呼び出しとして必ず `mcp__chrome-devtools__list_pages()` を実行する。これを実行せずに画像生成・画像保存・Pillowリサイズへ進むことは禁止**

## ChatGPT Images 2.0 生成プロトコル（DevTools経由）

Phase 6（表紙）・Phase 7（A+）の全画像生成で共通のプロトコルを使用する。
Codex CLI環境ではChrome DevTools MCPが `mcp__chrome-devtools__*` として利用可能。

### 前提条件

- `~/.codex/config.toml` に `[mcp_servers.chrome-devtools]` が設定済み（`--browserUrl http://127.0.0.1:9224` 固定）
- Chromeが `--remote-debugging-port=9224` で起動済み（未起動ならCodexが自動起動する）
- ChatGPTにログイン済み

### DevTools MCP 接続失敗時の対処（★最重要・絶対遵守）

```
┌─────────────────────────────────────────────────────────────────────┐
│  DevTools接続が失敗したら「別の方法で画像を作る」は絶対禁止。       │
│  何度でもDevToolsへの接続を試み続けること。                         │
│  代替手段を探す・提案する・検討すること自体が禁止。                 │
│  接続できるまで画像生成フェーズに進まない。                         │
└─────────────────────────────────────────────────────────────────────┘
```

Phase 6（表紙）・Phase 7（A+）の開始前に、必ず DevTools MCP の接続を確認する。

**Step A: 設定とChrome起動を自動確認**
  1. `~/.codex/config.toml` の chrome-devtools が `--browserUrl http://127.0.0.1:9224` になっているか確認
  2. 誤って `--port 9224` になっている場合は必ず `--browserUrl http://127.0.0.1:9224` に修正
  3. `http://127.0.0.1:9224/json/version` を確認
  4. 9224が未起動ならChromeを以下で自動起動:
     `"C:\Program Files\Google\Chrome\Application\chrome.exe" --remote-debugging-port=9224 --remote-allow-origins=* --user-data-dir="%USERPROFILE%\chrome-profiles\profile-9224" https://chatgpt.com`

**Step B: 自動リトライ（ユーザー操作不要・最大5回）**
  1. `mcp__chrome-devtools__list_pages()` を実行
  2. → 成功 → 通常の生成フローへ進む
  3. → 失敗 → 10秒待機 → 再試行（最大5回）
  ※ 5回すべて自動で試行する。1回失敗しただけで諦めない。

**Step C: 起動済みMCPが古い設定を掴んでいる場合の自動復旧**
  1. `mcp__chrome-devtools__list_pages()` が `Transport closed` または一時プロファイルを返す場合、古い `chrome-devtools-mcp --port 9224` プロセスと `~/.cache/chrome-devtools-mcp/chrome-profile` 側のChromeだけを停止する
  2. 通常Chromeの `%USERPROFILE%\chrome-profiles\profile-9224` は停止しない
  3. `codex exec --dangerously-bypass-approvals-and-sandbox -C "{repo}" "mcp__chrome-devtools__list_pages を実行してください"` で新しい実行を起動して再検証する
  4. → 成功 → 通常の生成フローへ進む

**Step D: ユーザー操作が必要な唯一のケース**
  ChatGPTがログアウト状態、Googleログイン拒否、二段階認証、CAPTCHA等を表示している場合のみ、ユーザーに通常Chrome上でログインを依頼する。
  認証情報の入力代行は禁止。
  ログイン後は Step B に戻る（**回数制限なし・永久ループ**）。

**このループはユーザーが「中止」「やめる」「スキップ」と明示的に言うまで絶対に終了しない。**
エージェントが自主判断で「接続できないので別の方法で」と切り替えることは禁止。

**絶対禁止（違反した場合は全画像の再生成が必要）:**
- DevTools接続失敗時に imagegen skill で画像生成すること
- DevTools接続失敗時に gpt-image-2 API で画像生成すること
- DevTools接続失敗時に DALL-E で画像生成すること
- DevTools接続失敗時に HTML/CSS/SVG で画像を描画すること
- DevTools接続失敗時に Pillow/PIL でテキスト描画すること
- DevTools接続失敗時に「代替手段」「別のアプローチ」を提案すること
- 「DevToolsが使えないため、テキストのみで完了とします」と報告すること

### 4ブロックプロンプト構造（全画像共通）

すべての画像プロンプトは以下の4ブロック形式で記述する:

```
* Subject: (画像の主題・内容。日本語テキストを含む場合はここに記載)
* Layout: (配置・構造・階層・接続関係の指示)
* Visuals: (色・アイコン・フォント・装飾の指示。カラースキームを明示)
* Style: (全体のスタイル・品質。末尾に必ず「Use GPT-5.5 for image generation. Do NOT use DALL-E or older models.」を付加)
```

### 生成フロー（Step 1-8）

```
Step 1: 接続確認
  │  mcp__chrome-devtools__list_pages()
  │  → エラー時: mcp__chrome-devtools__new_page(url="https://chatgpt.com") で復帰
  ▼
Step 2: ChatGPT Images 2.0 を新しいタブで開く
  │  mcp__chrome-devtools__new_page(url="https://chatgpt.com/g/g-2fkFE8rbu-dall-e")
  │  sleep 5（ページ読み込み待ち）
  ▼
Step 2.5: モデル確認（DALL-Eフォールバック防止）
  │  mcp__chrome-devtools__take_screenshot()
  │  → ページ上部のモデル表示を確認
  │  → 「4o」以外: モデル選択UIで「4o」を選択し直す
  ▼
Step 3: プロンプト入力（★日本語文字化け防止 — evaluate_script方式を使う）
  │
  │  ⚠ fill() は ChatGPT の ProseMirror 入力欄で日本語が文字化けするため使用禁止。
  │  以下の evaluate_script 方式で入力する:
  │
  │  mcp__chrome-devtools__evaluate_script(expression=`
  │    // ChatGPTの入力欄を特定（ProseMirror contenteditable）
  │    const editor = document.querySelector('#prompt-textarea, [contenteditable="true"]');
  │    if (editor) {
  │      editor.focus();
  │      // 既存テキストをクリア
  │      editor.textContent = '';
  │      // プロンプトをセット（ProseMirror用に<p>タグで囲む）
  │      const lines = PROMPT_TEXT.split('\\n');
  │      editor.innerHTML = lines.map(l => '<p>' + l + '</p>').join('');
  │      // Reactの状態同期のためinputイベントを発火
  │      editor.dispatchEvent(new Event('input', { bubbles: true }));
  │    }
  │  `)
  │
  │  ※ PROMPT_TEXT にはcover_prompt.txtの全文を文字列として埋め込む
  │  ※ プロンプト内のバッククォート(`)はエスケープする
  │  ※ evaluate_script でも文字化けする場合の代替:
  │     → mcp__chrome-devtools__click(入力欄uid) で入力欄にフォーカス
  │     → mcp__chrome-devtools__type_text(text=プロンプト) で1文字ずつ入力
  │     → type_text は遅いが文字化けしない最終手段
  │
  │  入力後、mcp__chrome-devtools__take_screenshot() で
  │  入力欄に日本語が正しく表示されていることを目視確認する
  ▼
Step 4: 送信
  │  mcp__chrome-devtools__take_screenshot()
  │  → 送信ボタンを確認
  │  → mcp__chrome-devtools__evaluate_script(expression=`
  │      document.querySelector('[data-testid="send-button"]').click();
  │    `)
  │  → evaluate_script で送信できない場合:
  │     mcp__chrome-devtools__take_snapshot() → 送信ボタンuid特定 → click(uid)
  ▼
Step 5: 生成待ち（ポーリング方式）
  │  sleep 30（初回待機）
  │  → mcp__chrome-devtools__take_screenshot() で完了確認
  │  → 未完了: sleep 15 → 再確認（最大4回）
  ▼
Step 6: モデル結果確認
  │  DALL-E生成の兆候チェック
  │  → 検知: close_page → 新タブで再生成
  ▼
Step 7: 画像保存
  │  方式A: ダウンロードボタンclick → ~/Downloads から移動
  │    sleep 3 → LATEST=$(ls -t ~/Downloads/*.png | head -1)
  │    mv "$LATEST" "output/{slug}/images/{filename}.png"
  │  方式B: evaluate_script で img src 取得 → curl で保存
  │  方式C: canvas → base64 → デコード保存
  ▼
Step 8: 次の画像 or 完了
  │  並列モード: new_page で別タブを開き Step 3 から並列
  │  逐次モード: 同じタブで Step 3 から繰り返し
  │  完了: close_page で不要タブを閉じる
```

### 複数枚生成時の注意事項

- 8タブ以上の同時生成はレートリミットの恐れ（6/2セッションで確認済み）
- 各タブは独立したChatGPTセッション（会話が混ざらない）
- 生成失敗時は該当タブを close_page → new_page で再生成
